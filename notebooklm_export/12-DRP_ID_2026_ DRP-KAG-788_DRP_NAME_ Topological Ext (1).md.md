The T5 Core: Monte Carlo Tree Search (MCTS)
MCTS doesn't require a GPU or cloud cluster — it requires a fast, accurate game simulator running in your local process within the 1-second turn budget. The architecture has four components:
1. The Forward Simulator A pure Python/NumPy reimplementation of the Orbit Wars turn order — comet expiration, fleet launch, production, fleet movement, planet rotation, combat resolution.  This is the hardest engineering task in the whole project. It must be exactly correct (especially the continuous collision detection against the sun and moving planets) and fast enough to run ~200–500 simulated turns per second. NumPy vectorization over planet/fleet arrays is the key — avoid Python loops over individual game objects.^7_1
2. The Tree Policy UCB1 (Upper Confidence Bound) is the standard selection formula:
\text{UCT} = \bar{X}_i + C \cdot \sqrt{\frac{\ln N}{n_i}}
where \bar{X}_i is the average reward from node i, N is the parent visit count, n_i is the node's visit count, and C is an exploration constant you tune empirically.  This balances exploring new action branches against exploiting known-good ones.^7_2
3. The Rollout Policy When you reach a leaf node, you simulate the game to completion using a fast "playout policy" — typically your T4 heuristic bot playing both sides. The quality of this rollout policy has a huge impact on MCTS signal quality. A random rollout policy converges slowly; a heuristic rollout policy converges much faster.^7_3
4. The Evaluation / Backpropagation At game end (or a depth limit), score the state — ship count differential is the natural metric. Backpropagate the reward up the tree.
